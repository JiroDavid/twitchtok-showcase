# Demo Tools Integration — Design Spec
_2026-06-01_

## Goal

Wire three preloaded Twitch clips into the demo landing page, integrate the existing crop and subtitle editor tools into the post-reveal step, and upgrade the subtitle editor with a live CSS preview overlay, timeline zoom, and audio waveform.

---

## 1. Clips & Demo Cache Structure

### Clip files
Three clips already downloaded to `frontend/public/clips/`:
- `clip1.mp4` — Ludwig
- `clip2.mp4` — JasonTheWeen
- `clip3.mp4` — StableRonaldo

User will provide cut versions of each clip. Cut clips replace the raw clips as both the picker preview source and the pipeline input. Raw files kept locally as backup.

### Demo cache
```
frontend/public/demo_cache/
  clip1/
    output.mp4        ← pre-rendered, manually corrected final video
    captions.json     ← [{id, start, end, final_text, style, placement}] (user provides)
    crop.json         ← saved after manual crop editor correction (generated by tool)
  clip2/  (same)
  clip3/  (same)
```

`captions.json` — user provides text + timings; we populate the JSON structure. Required fields per entry: `id` (integer), `start` (seconds, float), `end` (seconds, float), `final_text` (string). Optional: `style` (color, font_family, font_size, outline, shadow), `placement` (track, align). Matches `EditableCaptionDraft` shape from `types.ts`.
`crop.json` — produced by running the pipeline once then correcting via the crop editor; saved from the editor's output.
`output.mp4` — produced by running the pipeline once then correcting via both editors; the final corrected video is the demo output.

`demoClips.ts` updated with real `videoSrc` (`/clips/clip{n}.mp4`) and `inputPath` values plus correct titles, streamers, and durations.

## 2. Demo Flow: Pre-rendered Output

The `ProcessingWindow` component currently calls the backend. For the demo, it instead resolves to the pre-cached output after its timed animation completes.

**Change:** `ProcessingWindow` receives the clip index and resolves to `/demo_cache/clip{n}/output.mp4` when the animation finishes. No backend call for this step.

The MiniPhonePreview (live CSS font/colour preview during configuration) is unchanged — it is purely CSS-driven and unaffected by this change.

The backend still runs on the stand PC. The full app page (Twitch login, file input, real pipeline) remains fully functional for live demos.

## 3. RevealPanel — Fine-Tune Tools Section

Below the existing Download and "Try another clip" buttons, add a "Want to fine-tune?" section:

```
[ Download ]

─── Want to fine-tune? ───
[ ✏️ Edit Subtitles ]  [ ✂️ Adjust Crop ]
```

Both buttons open the existing modal tools. The section is always visible in the reveal step — no accordion, no hidden state.

**Props added to RevealPanel:**
- `clipIndex: number` — to load the correct cache files
- `onOpenSubtitleEditor: () => void`
- `onOpenCropEditor: () => void`

## 4. Demo Tool State (page.tsx)

`page.tsx` gains two new boolean states: `subtitleEditorOpen` and `cropEditorOpen`.

A `DemoRevealTools` component (or inline in `page.tsx`) manages:
- Loading `captions.json` for the selected clip when the subtitle editor opens
- Passing `outputUrl` as the subtitle editor's `outputVideoUrl`
- Passing the clip's `videoSrc` as the crop editor's `cropEditorPreviewUrl`
- Wiring the existing `onSave` / `onApply` / `onClose` callbacks

The existing `CropEditorModal` and `SubtitleEditorModal` components are used as-is — only the wiring into the demo flow is new code.

The crop editor requires drag state. A new `useDemoCropEditor` hook (in `components/demo/`) manages this: initialises crop boxes from `demo_cache/clip{n}/crop.json` on open, handles pointer drag events, and exposes `topPreviewStyle` / `bottomPreviewStyle` / `onStartDrag` / `onUpdateSplitRatio` / `onSave` — the exact props `CropEditorModal` expects. On save, the hook calls the backend re-render endpoint (same as the app page) and updates `outputUrl` in the demo stage.

## 5. Subtitle Editor — Live CSS Preview Overlay

**Current state:** The left panel shows the rendered output video as a reference. Changes only visible after "Apply Edits & Re-render".

**Change:** Overlay subtitle text as HTML elements directly on the video preview using absolute positioning, synced to `currentTime`.

Implementation:
- On each `timeupdate` event, find captions where `currentTime >= start && currentTime < end`
- Render active captions as absolutely-positioned `<div>` elements over the video
- Style each caption using `caption.style` (color, fontFamily, fontSize, outline, shadow) — same CSS approach as MiniPhonePreview
- Position using `caption.placement.track` (top/bottom/free) and `caption.placement.align`
- The overlay is a `pointer-events: none` div sitting above the video element

This is an approximation of the FFmpeg burn-in output (CSS ≠ exact pixel match) but accurate enough for editing purposes.

## 6. Subtitle Timeline — Zoom, Waveform & Resize

### Zoom
- Scroll wheel on the timeline container zooms in/out (1×–10× range)
- `+` / `−` buttons as fallback controls
- Zoom anchors to the playhead position
- When zoomed in, the timeline scrolls horizontally; a drag-to-pan gesture on the ruler
- `pixelsPerSecond` replaces the current percentage-based layout — caption block widths and positions are computed in pixels, the container scrolls

### Audio waveform
- On video load (`loadedmetadata`), decode audio via Web Audio API: `AudioContext.decodeAudioData`
- Downsample to ~200 data points across the clip duration
- Render as an SVG `<polyline>` or series of `<rect>` bars behind caption blocks, at ~25% opacity
- Color: `#9146ff` (matches the existing purple theme)
- Waveform is drawn once on load; no per-frame updates

### Resizable panel
- A drag handle (4px strip) on the top edge of the timeline section
- Dragging sets a `timelineHeight` state (min 80px, max 300px)
- The two track rows split the available height equally

## 7. Out of Scope

- Subtitle font picker changes — deferred; user to specify font preferences separately
- Audio waveform backend extraction — Web Audio API only, no server-side processing
- Live FFmpeg render for demo configuration choices (font/colour) — pre-rendered output used instead
- Any changes to the app page (`/app`) or existing modal logic beyond the live preview overlay and timeline improvements
